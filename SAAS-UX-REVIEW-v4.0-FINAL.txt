DEVON TORRES — DATA VISUALIZATION
────────────────────────────────────────────────────────────────────────────────

FOCUS: Dashboard design, metrics storytelling, data clarity.

PRESENT VIEW ASSESSMENT:

Strengths (8.5/10):
✓ Executive summary row (4 cards) is perfect information hierarchy
  → Initiatives (count), Plans (count), Health (%), At-Risk (count) = complete portfolio snapshot
  → No tooltip needed; metrics speak for themselves
✓ Horizontal health bars with percentage label: Excellent
  → Avoids chart junk; clear at 1 glance
  → Plan count badge (e.g., "8 plans") provides context
✓ Risk spotlight: Sorted by urgency (days remaining)
  → Shows owner + due date + initiative link
  → "Training Programs · 25 days" immediately tells me urgency
✓ Initiative detail cards: Health ring (SVG circle) + on-track count is sophisticated
  → The ring is elegant AND functional (visual + numeric)
  → Color gradient (teal→sage→gold→amber) tells story instantly

Concerns:
⚠ Portfolio Health: Calculated as AVERAGE across initiatives
  → Problem: If Initiative A has 10 plans (health 40%) and Initiative B has 1 plan (health 100%),
    the portfolio health is 70%. But operationally, Initiative A is much larger.
  → Consider WEIGHTED average by plan count, or show variance/distribution
⚠ Health metric doesn't distinguish between "healthy but slow" vs. "unhealthy fast"
  → A plan at 95% progress is "healthy" but might be ahead of schedule (risky)
  → A plan at 30% progress at month 2 of 6 (day 60 of 180) is on pace but shows as "low health"
  → Need progress-vs-timeline visualization, not just status
⚠ Risk Spotlight: Sorts by days remaining, but no "severity" dimension
  → A plan due in 5 days with 95% progress ≠ a plan due in 5 days with 20% progress
  → Consider sorting by "risk score" = (days_remaining / (days_total - days_elapsed)) × (100 - progress)
⚠ Sub-plan indentation in cards is subtle — looks like nesting but no obvious visual distinction
  → Consider bullet prefix (• vs. →) or colored left-border
⚠ No "sparkline" or trend visualization
  → Static snapshot of health; no sense of momentum (improving, declining, stable)
  → Show small trend icon: ↑ (improving) ↓ (declining) → (stable) next to health %

GARDEN HEADER ASSESSMENT:

Strengths:
✓ 4-card metrics layout matches Present view (consistency)
✓ Color coding (teal, sage-dark, health color, amber) matches data visualization standards
✓ Growth stage filter buttons show plan COUNT per stage
  → Data-driven filtering: "9 plans are Budding" = user knows the filter will show meaningful subset

Concerns:
⚠ "Total Plans: 38" doesn't break down by status
  → Metrics row would be stronger if it showed "38 plans (5 Seed, 12 Sprouting, 15 Budding, 6 Bloomed)"
  → Or dynamically: Click filter → metrics update to show filtered breakdown
⚠ "Portfolio Health" metric is redundant between Garden and Present views
  → Good for consistency, but becomes boilerplate
  → Consider: Garden shows AGGREGATE health; Present shows BY-INITIATIVE breakdown

OVERALL SCORE: 8/10
→ Overall layout and hierarchy are strong
→ Health calculation and trend visualization need sophistication
→ Data-driven filtering is good; needs metric breakdown

RECOMMENDATIONS:
1. Add trend indicator next to health %: ↑ +3%, ↓ -2%, → 0% (in 7 days)
2. Consider weighted health average: health = Σ(initiative_health × plan_count) / total_plans
3. Risk Spotlight: Sort by risk score (urgency + progress gap), not just days remaining
4. Break down "Total Plans" metric: Show "38 (5 Seed, 12 Sprouting, 15 Budding, 6 Bloomed)"
   OR: Update metrics dynamically when filter is active
5. Add progress-vs-timeline visualization: Small chart showing plan progress vs. expected progress
   Example: Plan A is 50% complete, but should be 67% at this point in timeline
6. Clarify sub-plan hierarchy: Use prefix (1 → 1.1 → 1.1.2) or emoji (🌱 🌿 🌼)

JAMES ADEYEMI — ACCESSIBILITY
────────────────────────────────────────────────────────────────────────────────

FOCUS: WCAG 2.1 AA compliance, keyboard navigation, screen reader support, cognitive load.

PRESENT VIEW ASSESSMENT:

Strengths (6.5/10):
✓ Semantic HTML: Headings (h1, h2, h3) are properly structured
✓ Color contrast: Most text passes WCAG AA (black text on white cards)
✓ Filter buttons: Have explicit aria-label (e.g., aria-label="Next month")
✓ Health ring SVG: Has role="img" + aria-label (good)

Concerns:
❌ CRITICAL: Metric cards lack semantic structure
  → Example: "Total Initiatives" is just a <div> with text "initiatives" — no <label>, no <aria-label>
  → Screen reader hears: "4" with no context — user doesn't know if this is initiatives, plans, health %, or risk count
  → FIX: Wrap in <section aria-label="Total Initiatives: 4"> or use <meter> or <dl> (definition list)

❌ CRITICAL: Risk Spotlight amber colors may fail contrast
  → Amber text (#C07A6B) on amber background (rgba(217, 119, 6, 0.05)) = ~2.1:1 contrast
  → WCAG AA requires 4.5:1 for text, 3:1 for large text
  → FIX: Darken amber text or lighten background; use white text with amber background instead

❌ Sub-plan hierarchy not semantically marked
  → Indentation visual ≠ semantic nesting
  → Screen reader reads "1.1 DGR Leadership" then "1.2 Workstation Normalization" as flat list
  → Visually nested, but semantically flat — screen reader user doesn't understand hierarchy
  → FIX: Use nested <ul> <li> or ARIA role="tree" with aria-expanded

⚠ Health ring SVG: aria-label says "Health 67%" but doesn't explain WHAT is healthy
  → Better: aria-label="Initiative Health: 67% (3 of 5 plans on track)"

⚠ Risk Spotlight sorting: "Sorted by days remaining" is not indicated to screen reader user
  → Sighted user sees items ordered by urgency; screen reader user hears unordered list
  → FIX: Add <caption> or <aria-describedby> explaining sort order

⚠ Filter buttons: Active state uses border + background color
  → But border-only active state (2px teal) may not be obvious to color-blind users
  → FIX: Add aria-pressed="true" and/or icon (✓) in addition to color change

⚠ No skip links or keyboard navigation hints
  → Tab order: Present view is long (metrics → health bars → risk spotlight → detail cards)
  → Users on keyboard must tab through 50+ elements
  → Consider: Skip to "Risk Spotlight" link or "Jump to [section]" navigation

⚠ Modal focus management: Exists in code (firstFocusRef, focus trap), but not all interactive buttons have focus-visible styles
  → Some buttons may not show focus outline on keyboard nav
  → Check: button:focus-visible { outline: 2px solid var(--teal); outline-offset: 2px; } (exists in CSS, good)

GARDEN HEADER ASSESSMENT:

Strengths:
✓ Filter buttons: Have click handlers + keyboard support (good)
✓ Color and icon used together (icon + text label = redundant encoding, accessible)

Concerns:
❌ Growth stage buttons: No aria-label or aria-describedby
  → Button text: "Seed" — OK for sighted user
  → Screen reader: "Seed" with no context of what it does or what count is
  → FIX: aria-label="Filter by Seed stage (5 plans)"

❌ "Clear filter" button: Only appears AFTER filter is active
  → But DOM is not changed; it's hidden with CSS or conditional rendering
  → If conditional, screen reader user must know to look for it
  → FIX: Always render button; disable or hide with aria-hidden when no filter is active

⚠ Metric cards: Same issue as Present view — no semantic labeling

OVERALL SCORE: 6/10
→ Basic accessibility is present, but semantic structure is weak
→ Color contrast issues in Risk Spotlight
→ Keyboard navigation and screen reader support need work
→ Cognitive load: Long-form cards + many details = overwhelming for users with cognitive disabilities

RECOMMENDATIONS (PRIORITY ORDER):

HIGH PRIORITY:
1. Wrap metric cards in <section aria-label="[Metric Name]: [Value]">
   Example: <section aria-label="Portfolio Health: 67%">
2. Fix Risk Spotlight contrast: Use white text on amber background OR dark amber text on white background
3. Add semantic nesting to sub-plan hierarchy:
   <ul>
     <li>1.1 DGR Leadership <ul><li>1.1.1 Internal workflows...</li></ul></li>
   </ul>
   OR use role="tree" + aria-expanded
4. Add aria-pressed="true/false" to filter buttons + visual indicator (icon, not just color)
5. Add aria-label to growth stage filter buttons: "Filter by [Status] stage ([count] plans)"

MEDIUM PRIORITY:
6. Improve health ring aria-label: "Initiative Health: 67% (3 of 5 plans on track)"
7. Add sort order explanation to Risk Spotlight: <caption> or <aria-describedby>
8. Ensure button:focus-visible styles are visible (they are in CSS, but test on actual device)
9. Add skip links or section navigation (e.g., "Skip to Risk Spotlight", "Jump to Initiative Details")
10. Test color contrast with WebAIM contrast checker (https://webaim.org/resources/contrastchecker/)

LOW PRIORITY:
11. Add ARIA descriptions to help users with cognitive disabilities parse information
    Example: <h2>Initiative Health Overview</h2><p>Each bar shows health of an initiative (0-100%). Green = healthy, red = at-risk.</p>
12. Consider: Add print-to-speech button for accessibility
13. Test with screen reader (NVDA, JAWS, VoiceOver) — may reveal additional issues

TOP 5 ISSUES IDENTIFIED (Consensus):

1. **ACCESSIBILITY BLOCKER: Semantic structure**
   - Metric cards, risk spotlight, sub-plans lack semantic nesting
   - Affects: Screen reader users, users relying on assistive tech
   - Severity: HIGH (WCAG 2.1 AA non-compliance)
   - Scope: Present view + Garden header

2. **CONTRAST FAILURE: Risk Spotlight amber on amber**
   - Amber text on amber-light background = 2.1:1 contrast (fails WCAG AA 4.5:1)
   - Affects: Users with low vision, color-blind users
   - Severity: HIGH (WCAG non-compliance)
   - Scope: Present view Risk Spotlight section

3. **DATA INSIGHT GAP: No trend/context on health metrics**

   - Total plans doesn't break down by status

   - Severity: MEDIUM (reduces decision-making value)
   - Scope: Present view + Garden header metrics

4. **MOBILE UX: Touch target size + filter feedback**
   - Filter buttons < 48px on mobile
   - Active filter state not obvious (border only, subtle on small screens)
   - Severity: MEDIUM (affects ~40% of users, reduces usability on mobile)
   - Scope: Garden view, filter buttons

   - Visually indented + border-left, but not semantically nested
   - Screen readers read as flat list (1.1, 1.2, 1.3 all at same level)

  // At top of main App component, add state:
  const [prevHealth, setPrevHealth] = useState(null);
  const [lastHealthUpdate, setLastHealthUpdate] = useState(null);

  // When health calculation changes:
  useEffect(() => {
    const now = new Date();
    const savedUpdate = localStorage.getItem('lastHealthUpdate');
    const savedHealth = localStorage.getItem('prevHealth');
    
    if (lastHealthUpdate && lastHealthUpdate.toDateString() === now.toDateString()) {
      // Already updated today; do nothing
    } else {
      // New day; save today's health as prevHealth for tomorrow
      setPrevHealth(portfolioHealth);
      setLastHealthUpdate(now);
      localStorage.setItem('prevHealth', portfolioHealth);
      localStorage.setItem('lastHealthUpdate', now.toISOString());
    }
  }, [portfolioHealth]);

  // In Present view metric card:
  {/* Portfolio Health */}
  <div style={{ ... }}>
    <div style={{ fontSize: '12px', ... }}>PORTFOLIO HEALTH</div>
    <div style={{ display: 'flex', alignItems: 'baseline', gap: '8px' }}>
      <div style={{ fontSize: '32px', fontWeight: 700, color: healthColor(portfolioHealth) }}>
        {portfolioHealth}%
      </div>
      {prevHealth !== null && (
        <div style={{
          fontSize: '12px',
          fontWeight: 600,
          color: portfolioHealth > prevHealth ? '#2A6B6B' : portfolioHealth < prevHealth ? '#C07A6B' : '#7A716A'
        }}>
          {portfolioHealth > prevHealth && `↑ +${portfolioHealth - prevHealth}%`}
          {portfolioHealth < prevHealth && `↓ -${Math.abs(portfolioHealth - prevHealth)}%`}
          {portfolioHealth === prevHealth && `→ No change`}
